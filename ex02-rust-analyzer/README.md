# Exercise 02 ‚Äî rust-analyzer + a no_std ecosystem crate

15 minutes. Two payoffs:

1. **Make rust-analyzer work** inside an exercise directory using
   the `.cargo/config.toml` symlink trick the lecture just covered.
2. **Add a `no_std` ecosystem crate** (`heapless`) as a real Cargo
   dependency, use it from `rust_main`, and watch it just work.

The starter is an unmodified copy of `samples/hello_world` ‚Äî
`Cargo.toml`, `CMakeLists.txt`, `prj.conf`, `sample.yaml`,
`Cargo.lock`, and `src/lib.rs`.

## 1. Build the starter

This produces the cargo-config template that the symlink will
point at:

```
cd roz-workshop/ex02-rust-analyzer
west build -b qemu_cortex_m3 .
```

It should build and link cleanly. Don't run it yet ‚Äî we'll modify
the source first.

## 2. Apply the cargo-config symlink

Follow the recipe in [`../common/README.md`](../common/README.md):

```
mkdir -p .cargo
ln -sf ../build/rust/sample-cargo-config.toml .cargo/config.toml
```

The symlink lives in your working copy, not in git
(`.cargo/config.toml` is in the repo's `.gitignore`).

## 3. Confirm `cargo check` works

```
cargo check
```

It should reach `Checking rustapp` and finish successfully.

## 4. Open the exercise in your editor

Open `roz-workshop/ex02-rust-analyzer/` in whatever editor you use
(VS Code, Helix, neovim with rust-analyzer, JetBrains, etc.).
Confirm rust-analyzer is happy:

- Hover types in `src/lib.rs`.
- Jump-to-definition on `info!` or `zephyr::set_logger`.
- Type `zephyr::` somewhere and watch autocomplete come up.

If rust-analyzer is silent, the symlink in step 2 didn't take ‚Äî
re-check it before moving on.

## 5. Add a real Cargo dependency

Open `Cargo.toml` and add `heapless` under `[dependencies]`:

```toml
[dependencies]
zephyr = "0.1.0"
log = "0.4.22"
heapless = "0.8"
```

Run `cargo check` again. Cargo will fetch `heapless` and its
deps, type-check the tree, and finish.

## 6. Use `heapless::String` from `rust_main`

Edit `src/lib.rs`. Build a message into a stack-allocated
`heapless::String` and pass it through `printk!`:

```rust
#![no_std]

use core::fmt::Write;
use heapless::String;
use log::info;
use zephyr::printk;

#[no_mangle]
extern "C" fn rust_main() {
    unsafe {
        zephyr::set_logger().unwrap();
    }

    info!("Hello world from Rust on {}", zephyr::kconfig::CONFIG_BOARD);

    let mut msg: String<32> = String::new();
    write!(&mut msg, "hello from rust-on-zephyr").unwrap();
    printk!("{}\n", msg);
}
```

Two things to notice:

- `heapless::String<32>` is a fixed-capacity, stack-allocated
  string ‚Äî no allocator needed. Its const generic parameter is
  the byte capacity.
- `printk!` takes the same `format_args!` syntax as `println!`
  in std, and it accepts our `heapless::String` because that type
  implements `Display`.

## 7. Rebuild and run

```
west build -b qemu_cortex_m3 .
west build -t run
```

You should see the `info!` line and the `printk!` line under QEMU.
Press `Ctrl-a x` to exit.

## Punchline

Adding a `no_std` ecosystem crate to a Rust-on-Zephyr application
is the same Cargo workflow you'd use anywhere else: edit
`Cargo.toml`, write the code, build. No platform-specific dance,
no special build flags, no hand-rolled glue. The wider `no_std`
Rust ecosystem comes along for the ride ‚Äî `heapless`, `bitvec`,
`embedded-hal`, `serde` (in `no_std` mode), `postcard`, `defmt`,
and so on.

This is one of the genuine wins of Rust-on-Zephyr over either
plain Zephyr-C (no Cargo) or a more minimal embedded-Rust HAL
(no Zephyr services).

## If you change boards or wipe `build/`

Re-run the symlink step (`ln -sf ../build/rust/sample-cargo-config.toml .cargo/config.toml`).
The template gets regenerated by cmake; the symlink target is
the same path but a fresh file underneath.
